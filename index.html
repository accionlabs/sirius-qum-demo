What QUM is, exactly

Quantitative User Metrics is a framework that converts a small number of authored user scenarios into a dense set of measured, comparable numbers — so user experience can be discussed with evidence rather than anecdote.

Input

Authored scenarios

A scenario is a named user journey (e.g. Transportation-based project search) expressed as a sequence of activities, steps, and actions. Each action is an executable Nexial command that drives a real browser through real UI.

Execution

Instrumented playback

As Nexial executes the scenario, it captures performance timings (TTFB, DOM interactive, page complete), accessibility violations against WCAG A/AA/AAA, design hygiene signals, and the full user-action / system-delay / network profile at every step.

Output

Four linked views

Accessibility, QUM Performance, Browser Performance, and Features — each rolled up per persona. Sprint-over-sprint trends allow regression detection when the same scenarios are replayed after changes.

Why this matters for Sirius

Phil's Accelerate plan for Hubexo calls for MTT (mean-time-to) metrics across the SDLC. QUM supplies the user-facing half of that baseline: before an agent rewrites a screen, we know precisely what the user experiences today. After the rewrite, replaying the same scenarios shows whether the change actually moved the numbers.

What gets measured

Four metric families, all captured from the same scenario run

Family What it captures Example signals Used to answer
Accessibility WCAG 2.1 violations observed on every visited screen WCAG A / AA / AAA counts, element-level issues How compliant is the experience? Where are the blockers for assistive tech?
QUM Performance User-facing interaction cost across the journey User-action time, system delay, network-call volume How much of the task time is the user waiting on the system?
Browser Performance Low-level page load characteristics at each navigation TTFB, DOM interactive, page complete, content download Is slowness in the network, the server, or the client rendering?
Features The scenario itself — activities, steps, actions Pass/fail per step, click-paths, design hygiene flags Is the journey actually completable? Which step is the most fragile?

What the numbers suggest

  • Perceived latency dominates. System delay slightly exceeds user-action time — the user waits on the app more than they drive it.
  • Post-login is the slow moment. The initial click after login takes ~12 seconds of backend time in a single step.
  • Accessibility debt is real. 42 combined WCAG A/AA violations on a journey that only touches four screens.
  • Chatty search path. 483 network calls for a three-step journey signals an opportunity for request consolidation or client-side caching.

About this demo site

A static site hosted on GitHub Pages, wrapping the QUM report with the context a first-time reader needs. The report bundle itself is the original build output — only the landing context has been added.

Scope

Showcase-only. All figures come from a single scenario run on a live staging of Lead Manager during the Sirius assessment. A production QUM baseline would cover multiple personas and dozens of scenarios.

Reproducibility

The underlying scenario is a Nexial .xlsx script (searchForProject.xlsx). Re-running the script produces a new dated folder under /report/ and the summary pointer file is updated so the report always opens the latest run.
